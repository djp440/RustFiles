# 梳理文件管理器V1需求

> [$cdd-brainstorming](C://Users//15575//.agents//skills//cdd-brainstorming//SKILL.md)
>
> # 产品定位
>
> **一款面向 Windows 用户的高性能、精致、现代化文件管理器。**
>
> 关键词：
>
> **快、稳、漂亮、统一、顺手、可替代资源管理器。**
>
> 技术假设：
>
> * 后端/核心：**Rust**
> * 前端/UI：**React**
> * 桌面壳：优先考虑 **Tauri**
> * 目标系统：**Windows 10 / 11**
>
> ---
>
> # V1 核心目标
>
> V1 只打四件事：
>
> 1. **打开目录极快**
> 2. **文件浏览体验优雅**
> 3. **基础文件操作稳定可靠**
> 4. **UI 明显比 Windows 资源管理器高级**
>
> ---
>
> # V1 必做功能
>
> ## 1. 文件浏览
>
> 必须支持：
>
> * 本地磁盘浏览
> * 文件夹进入/返回/前进
> * 地址栏路径跳转
> * 面包屑路径导航
> * 侧边栏：
>
>   * 快速访问
>   * 桌面
>   * 下载
>   * 文档
>   * 图片
>   * 视频
>   * 音乐
>   * 此电脑
>   * 各磁盘分区
> * 文件列表：
>
>   * 图标视图
>   * 列表视图
>   * 详细信息视图
> * 排序：
>
>   * 名称
>   * 修改时间
>   * 大小
>   * 类型
> * 基础过滤：
>
>   * 文件
>   * 文件夹
>   * 图片
>   * 视频
>   * 文档
>
> ---
>
> ## 2. 文件操作
>
> V1 必须做稳：
>
> * 新建文件夹
> * 重命名
> * 删除到回收站
> * 永久删除
> * 复制
> * 剪切
> * 粘贴
> * 拖拽移动
> * 拖拽复制
> * 多选
> * 右键菜单
> * 打开文件
> * 在默认应用中打开
> * 在终端中打开当前目录
> * 显示属性
>
> 文件冲突处理：
>
> * 替换
> * 跳过
> * 保留两者
> * 应用于全部
>
> ---
>
> ## 3. 性能体验
>
> 这是核心卖点。
>
> V1 性能目标建议：
>
> * 冷启动：**≤ 800ms**
> * 普通文件夹打开：**≤ 100ms**
> * 1 万文件目录初次渲染：**≤ 500ms 可交互**
> * 列表滚动：稳定 **60fps**
> * 文件操作不能卡 UI
> * 大目录必须使用虚拟列表
> * 缩略图异步加载
> * 文件搜索/扫描不能阻塞主线程
>
> 关键实现：
>
> * Rust 负责文件系统访问、索引、缩略图、复制任务
> * React 只负责 UI 状态和渲染
> * 文件操作任务队列化
> * 前端列表虚拟化
> * 后台线程池处理 IO
>

V1 成功标准写成：启动速度 ≤ 800ms，大目录 ≤ 500ms 可交互，滚动 60fps，后台任务不影响前台体感，基础文件操作失败率极低且可恢复，UI 设计语言统一，内测崩溃率低于 1%，用户能日常替代 Windows 资源管理器。

1. **V0.1 浏览骨架**：应用窗口、侧边栏、路径栏、面包屑、文件列表、本地目录进入/返回。
2. **V0.2 视图与排序**：图标/列表/详细信息视图、排序、基础过滤、显示隐藏文件/扩展名。
3. **V0.3 性能底座**：虚拟列表、大目录加载、异步扫描、UI 优先调度、滚动流畅性。
4. **V0.4 搜索**：当前目录文件名搜索、实时过滤、递归开关、打开所在位置。
5. **V0.5 文件操作基础**：新建文件夹、重命名、删除到回收站、永久删除、打开文件、默认应用打开、属性。
6. **V0.6 复制移动闭环**：复制、剪切、粘贴、拖拽移动/复制、任务队列、进度反馈、失败恢复。
7. **V0.7 冲突处理与多选**：冲突替换/跳过/保留两者/应用全部、多选、右键菜单完善。

推荐采用 **方案 B：分层任务驱动架构**。

**方案 A：简单命令式架构**
React 每次操作直接调用 Tauri command，Rust 同步或短任务返回结果。优点是 V0.1 很快；缺点是复制/移动/缩略图/搜索一多，状态和取消会散，后期容易卡 UI。

**方案 B：分层任务驱动架构（推荐）**
React 管 UI 状态和用户意图；Rust 分成文件系统服务、导航/扫描服务、任务队列、缩略图服务、设置服务、系统集成服务。所有写操作进入 Rust 任务队列，读操作可分页/流式返回，缩略图和搜索按优先级调度。它最匹配“UI 优先调度”和 R4 文件安全边界。

**方案 C：索引优先架构**
先建本地索引/数据库，再让 UI 主要读索引。搜索和大目录体验会很好，但 V1 不做全盘索引，复杂度偏高，容易偏离首发闭环。

**架构方向草案**

- **模块边界**：React/Tauri UI 层不碰真实文件写入；Rust Core 是唯一文件系统副作用入口。
- **核心 Rust 模块**：`fs` 文件浏览与元数据、`tasks` 文件操作任务队列、`scheduler` UI 优先调度、`thumbnails` 缩略图、`search` 当前目录搜索、`settings` 设置、`system` 回收站/默认应用/终端/属性。
- **数据流**：前端发起“浏览/搜索/操作意图” -> Tauri command -> Rust 服务 -> 返回快照或任务 id -> Rust 通过事件推送进度、增量结果、错误。
- **状态归属**：标签页路径、视图、排序、搜索输入归前端；真实文件任务状态、取消、冲突、重试归 Rust；设置持久化由 Rust 管理，前端缓存展示。
- **接口契约**：读接口支持分页/排序/过滤；写接口只接受结构化操作请求并返回任务 id；冲突通过任务暂停状态等待用户选择。
- **失败处理**：所有破坏性操作先在隔离任务模型中记录阶段；失败返回可恢复状态。永久删除必须二次确认，覆盖/移动/冲突处理必须可审计。
- **e2e 验证**：后续用隔离测试目录生成小目录、大目录、冲突目录、权限/占用场景；用桌面运行证据 + 文件系统断言 + 前端交互截图/日志验证。
- **主要取舍**：V1 不引入全盘索引或数据库型文件目录索引；只为设置、缩略图缓存和任务历史保留轻量持久化空间。优先保证文件安全、UI 体感和可测性。

> | `cancelling`                    | 已请求取消，等待安全取消点 |
> | `cancelled`                     | 已取消           |
> | `completed`                     | 完成            |
> | `failed`                        | 失败，无进一步执行     |
> | `partially_completed`           | 部分完成，需要用户可见提示 |
>
> 这对复制/移动/删除尤其重要。CDD 要求不能只验证成功路径，必须覆盖冲突、权限、占用、取消或失败恢复。
>
> ### 4. UI 优先调度还需要可执行的信号和预算
>
> `scheduler` 方向对，但目前还偏抽象。建议补：
>
> * 前端向 Rust 上报：`active_tab_id`、visible range、最近输入时间、滚动状态。
> * Rust 侧优先级：前台目录枚举 > 可视缩略图 > 搜索增量 > 非可视缩略图。
> * 事件背压：`thumbnail_ready`、`search_result_batch` 必须批量合并，避免事件风暴。
> * 性能预算：冷启动、目录打开、10k 文件首屏、滚动帧时间的采样边界。
>
> PRD 对性能有明确数值：冷启动 <= 800ms、普通文件夹 <= 100ms、1 万文件目录 <= 500ms 可交互、滚动稳定 60fps。架构文档已经列了指标，但还需要定义“从哪里开始计时，到哪里结束”。
>
> ### 5. `DirectoryPage.snapshot_version` 需要补一致性规则
>
> 现在有快照版本，但没写清楚：
>
> * 目录变化后旧快照是否失效。
> * 排序/过滤变化是否产生新 snapshot。
> * 前端缓存什么时候必须刷新。
> * 文件操作完成后刷新哪些目录。
> * 搜索结果打开所在位置时，如果文件已经不存在，如何处理。
>
> 这会直接影响多标签和后台任务。建议给 `directory_changed` 增加：`path`、`old_snapshot_version`、`reason`、`suggested_refresh_scope`。
>
> ### 6. Tauri 权限边界建议写成显式白名单
>
> 文档说 Tauri Shell 控制前端可调用命令白名单，这是对的，但建议更具体：
>
> * 只暴露架构列出的 command。
> * 永久删除、覆盖、移动、冲突 resolve 必须带确认字段或确认令牌。
> * 测试模式下危险 command 必须经过统一 guard。
> * 禁止前端使用 Tauri 文件系统插件直接写真实文件。
>
> 这条很重要，因为“React 不具备真实文件写入权限”不能只停留在设计原则里，要落到 Tauri capability / command 暴露策略上。
>
> ## 建议补充的小项
>
> * `Settings` 增加 `schema_version`，并要求原子写入，避免设置文件写坏。
> * 缩略图缓存补充 key 策略：路径 + mtime + size，或内容 hash；还要有最大缓存体积和清理策略。
> * `FileEntry` 建议区分展示路径、真实路径、脱敏路径，避免日志里泄露完整用户目录。
> * `system` 模块建议补错误码：回收站不可用、默认应用打开失败、终端不可用、属性窗口打开失败。

我建议这样定方向：

**主风格：Apple-like Liquid Workbench**
吸收 Apple UI 的清晰、内容优先、层级、直接操作感和反馈可恢复原则。整体是安静、清爽、精致、有呼吸感的高端文件工作台。

**玻璃实现策略：**
- **窗口底层**：Tauri 发布版优先使用 Windows 系统级 Acrylic / Mica / Blur 类窗口效果。Tauri v2 window effects 支持 `Acrylic` / `Blur`，但 Windows 上效果与性能受系统版本影响。

- **内容区克制**：文件列表本体不能大面积透明到影响可读性。列表区域应使用更实的半透明表面，保证文件名、大小、时间、图标在浅色/深色下都清楚。
- **降级方案**：必须支持关闭透明度/降低透明度；不支持 `backdrop-filter` 或性能不足时退回实色 surface。
- **性能约束**：滚动大目录、缩略图加载、拖拽选择时，玻璃效果不能导致掉帧。必要时滚动期间降低非关键模糊层。

> 现在键盘部分覆盖了基础焦点和打开选择，但还缺文件管理器高频快捷键。建议补：
>
> | 快捷键                  | 行为           |
> | -------------------- | ------------ |
> | `F2`                 | 重命名          |
> | `Delete`             | 删除到回收站       |
> | `Shift + Delete`     | 永久删除确认       |
> | `Ctrl + C / X / V`   | 复制 / 剪切 / 粘贴 |
> | `Ctrl + A`           | 全选           |
> | `Ctrl + L`           | 聚焦地址栏        |
> | `Alt + Left / Right` | 后退 / 前进      |
> | `Ctrl + T`           | 新建标签页        |
> | `Ctrl + W`           | 关闭标签页        |
> | `F5`                 | 刷新当前目录       |
>
> V1 要替代资源管理器，这些会非常影响“顺手”的体感。
>
> ### 3. 多选和 inline rename 行为还不够细
>
> 文件管理器里最容易出细碎 bug 的就是选择模型。建议补：
>
> * `Ctrl` 多选。
> * `Shift` 范围选择。
> * 鼠标框选 / marquee selection 是否进入 V1。
> * 点击空白区是否清除选择。
> * 右键未选中项时，是先选中该项还是保留原多选。
> * inline rename 的提交、取消、重名错误、非法字符错误。
> * 重命名输入框是否保持 item 尺寸不跳动。
>
> 这些不一定都要在设计文档里写很细，但至少要给实现计划输入。
>
> ### 4. 玻璃性能降级需要触发规则
>
> 文档已经写了“大目录滚动期间可以降低非关键玻璃模糊强度”，这是对的。但建议补成更可执行的规则：
>
> * 滚动中关闭或降低 `content` blur。
> * 连续帧时间超过阈值时降低非关键 blur。
> * 停止滚动 150-300ms 后恢复。
> * 降低透明度模式永远不用 blur。
> * 低端 GPU / 远程桌面环境使用 solid fallback。
>
> 否则实现时容易变成“理论上可以降级，实际上没人知道什么时候降级”。
>
> ### 5. 冲突处理弹窗需要补批量冲突流程
>
> 现在冲突弹窗写了源 / 目标 / 时间 / 大小 / 缩略图，以及替换、跳过、保留两者、应用于全部。方向对，但批量复制移动时还需要定义：
>
> * 多个冲突是逐个弹，还是列表式批量处理。
> * “应用于全部”是应用于所有冲突，还是同类型冲突。
> * 保留两者的新名称预览是否每个冲突都显示。
> * 用户取消时，任务进入 `cancelled` 还是 `partially_completed`。
> * 冲突弹窗关闭按钮是否等价于取消当前任务。
>
> 建议这里补一段，因为它和 Rust 任务状态机直接相连。
>
> ### 6. 设置保存语义需要明确
>
> 设置页写了“切换设置、保存、恢复默认”，但没有明确是：
>
> * 改一项立即保存；
> * 还是用户点“保存”后统一提交；
> * 保存失败后回滚 UI；
> * 还是显示“未保存更改”。
>
> 建议 V1 用简单策略：**大多数设置即时保存，路径类/高成本设置可以显式确认；所有设置必须等 Rust 确认后才显示持久化成功。**
>
> 这和架构里的 settings 模块更一致。
>
